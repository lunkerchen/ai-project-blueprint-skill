# AI Project Blueprint — AI 專案藍圖

**把泛用 AI 助理打造成客製化 AI 員工。**

一套 6 步驟框架，用於設定高效能的 AI Agent 專案 — 改編自 Khairallah AL-Awady 的 Claude Projects 方法論。經過三週實測、12 次重構，產出 consistently excellent 的 blueprint。

> 多數人把 AI project 當成有標籤的資料夾。  
> 高手把它們打造成客製化 AI 員工 — 每個 project 在對話開始前就知道你的語氣、標準、受眾、工作流程，以及你對「好 output」的定義。

---

## 為什麼多數 AI Project 失敗

1. **System prompt 太模糊** — "You are a helpful assistant" 沒有告訴 AI 任何它不知道的事。它不知道的是你的具體語氣、受眾、格式規則、品質標準、以及你討厭的十五件事
2. **沒有 knowledge files** — 沒有知識庫的 project 只是標籤對話
3. **一個 project 想包辦所有事** — 把所有工作塞進同一個 project，結果全部平庸。每個 project 專注一件事

---

## 6 步驟藍圖

每個高效能 project 都有六個元件。跳過任何一個，輸出品質都會明顯下降。

### 1. 身份定義 (Identity Block)
System prompt 的第一段。告訴 AI 它是誰。

**錯誤：** `You are a helpful writing assistant.`

**正確：**
```
你是我的資深內容策略師兼寫手。你跟我合作了兩年，完全了解我的語氣、受眾和品質標準。

你在這個 project 的工作是為 25 萬名 AI 初學者讀者研究、規劃、撰寫和編輯長篇文章。

你直接、具體、絕不灌水。每一句話都必須證明自己值得存在，否則刪掉。
```

### 2. 規則區塊 (Rules Block)
最重要的部分。一組不可協商的 ALWAYS / NEVER 清單。

```
ALWAYS:
- 使用短段落。每段最多 3 句。
- 用具體數字取代模糊宣稱。
- 每節的關鍵洞察要加粗。

NEVER:
- 使用「賦能」「優化」「賦予」等空泛詞彙
- 以「現今」「值得注意的是」開頭
- 寫無法套用到任何人身上的泛泛建議
```

**祕訣：** 每次你發現自己在修正 AI 的輸出，那次修正就成為一條新的 rule。

### 3. 流程區塊 (Process Block)
定義 AI 如何思考，而非產出什麼。

```
流程：
1. 寫之前先想：讀者已經相信什麼？開頭必須挑戰那個信念。
2. 先列大綱再寫任何段落。
3. 第一稿一氣呵成，不要邊寫邊改。
4. 寫完後對照 rules 清單檢查，修正違規。
5. 確認每個宣稱都有具體數字或例子支持。
```

### 4. 輸出格式區塊 (Output Format Block)
消除對最終成品的所有猜測空間。

```
輸出格式：
- 標題：[數字] + [具體主題] + [排除鉤子]
- 長度：2000-3500 字
- 結構：對比開場 > 背景脈絡 > 編號主內容 > 總結行動呼籲
- 副標題：加粗，寫得像鉤子
```

### 5. 知識庫 (Knowledge Files)
這是 project 獲得真正智慧的地方。

**必備：**
- **風格指南** — 至少 3 篇最佳作品，讓 AI 學會你的語氣
- **受眾檔案** — 讀者是誰？知道什麼？想要什麼？

**建議：**
- 競爭者分析
- 歷史表現數據
- 可重用模板庫

### 6. 入職訊息 (Onboarding Message)
每次新對話的第一條訊息 — 就像每天給員工的 briefing。

```
今天要做的是：[任務]

脈絡：[要做什麼、為什麼]

風格參考我的指南。受眾參考我的受眾檔案。

開始之前先回答：
1. 這篇內容跟現有文章比，差異點在哪？
2. 讀者原本相信什麼，我們可以挑戰什麼？
3. 一句話的 hook 是什麼？
```

## 完整 System Prompt 模板

```
身份：
你是我的[角色]。你跟我合作了兩年，完全了解我的語氣、受眾和標準。

你在這個 project 的工作是[主要功能]，為[受眾描述]服務。

你[溝通風格]。你絕不[反模式]。

規則：
一定要：
- [規則 1]
- [規則 2]

絕不能：
- [反規則 1]
- [反規則 2]

流程：
1. [步驟 1 - 思考/規劃]
2. [步驟 2 - 大綱/結構]
3. [步驟 3 - 執行]
4. [步驟 4 - 根據規則審查]
5. [步驟 5 - 品質檢查]

輸出格式：
- 格式：[精確規範]
- 長度：[字數範圍]
- 結構：[段落順序]
```

## 你需要的 5 個 Project

| Project | 重點 | 核心知識庫 |
|---------|------|-----------|
| **內容創作** | 寫作語氣、受眾、標準 | 風格指南、代表作、受眾 persona |
| **研究分析** | 研究結構、來源信任 | 研究方法、過往分析、評估標準 |
| **溝通協作** | email/訊息風格、專業語氣 | 優秀範例、常用模板 |
| **策略規劃** | 商業脈絡、目標、限制 | 商業計畫、競爭者資料、策略框架 |
| **程式開發** | 技術棧、編碼慣例 | CLAUDE.md、ADR、範例程式碼 |

---

## 預期效果

- **設定前：** 詳細 prompt → 還可以的答案 → 花 20 分鐘編輯 → 每天 5-10 次
- **設定後：** 兩行任務描述 → 已經符合你的語氣、規則、受眾、品質標準的輸出 → 花 2 分鐘審查 → 每天 5-10 次

**減少 90% 編輯時間。** 一次性設定約 45 分鐘，每年省 250+ 小時。

## 套用到 Hermes Agent

這套 blueprint 可直接映射到 Hermes 生態：

- **身份 + 規則 →** `AGENTS.md` / profile system prompt
- **流程 + 輸出格式 →** Skill 的 `SKILL.md` 步驟與格式規範
- **知識庫 →** Skill 的 `references/` 目錄、memory 中的範例
- **入職訊息 →** 每次新任務的第一條指令，引用相關 skills
- **5 個 Project →** 各自的 Hermes profiles，各有自己的 skills/config
